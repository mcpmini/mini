### WEBAPP-441 — JWT `alg:none` bypass fix

**Jira:** WEBAPP-441 (Highest priority, In Progress)
**Sentry:** AUTH-001 — 2,847 fatal hits in production via `POST /auth/token`

**Root cause** (`auth/middleware.go:29`): The `ParseWithClaims` key function returned the signing key unconditionally, never inspecting `token.Method`. The `golang-jwt` library skips signature verification when the header declares `alg: none`, so attackers could forge arbitrary claims with no signature at all.

**Fix** (`auth/middleware.go:31-33`): Added a type assertion on `*jwt.SigningMethodHMAC` inside the key function. Any token whose algorithm is not an HMAC variant (HS256/HS384/HS512) — including `alg: none` and `alg: RS256` — is now rejected before the signature is evaluated.

```go
if _, ok := token.Method.(*jwt.SigningMethodHMAC); !ok {
    return nil, fmt.Errorf("unexpected signing method: %v", token.Header["alg"])
}
```

## WEBAPP-441 — JWT `alg:none` bypass fixed

**Jira:** WEBAPP-441 (Priority: Highest, CVE-2026-1337)
**Sentry:** AUTH-001 — 2,847 fatal occurrences in `auth/middleware.go:ValidateToken`

### Root cause

`jwt.ParseWithClaims` was called with a key function that returned the signing key unconditionally, without checking the token's algorithm. A forged token with `alg: none` passes through signature verification because the library skips verification for the `none` algorithm.

### Fix (`auth/middleware.go:32`)

Added a whitelist check inside the key function — if the signing method is not `jwt.SigningMethodHS256`, an error is returned before the key is ever handed back:

```go
if token.Method != jwt.SigningMethodHS256 {
    return nil, fmt.Errorf("%w: unexpected signing method %q", ErrInvalidToken, token.Header["alg"])
}
```

Rejection happens at parse time, before any claims are trusted.
